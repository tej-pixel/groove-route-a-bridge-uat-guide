Route A Collection Bridge — UAT Walkthrough

A step‑by‑step, screenshot‑backed guide for testing the Chargebee → Revenue Cloud Billing collection bridge end‑to‑end in the Groove sandbox — no data migration required. Every screenshot below is a real record produced by the bridge during sandbox testing.

Org  Groove_Sandbox
Gateway  Authorize.net Test (sandbox)
Audience  SF team · Finance/AR
Prepared by  SaaScend

Overview

When an RCB Invoice is Posted and in‑scope (no Chargebee ID, balance > 0), an asynchronous flow creates a Chargent Charge Order linked to the invoice and then either:

  • Auto‑collect customers (Account.Auto_Collection__c = true) → the stored card token is charged automatically.
  • Invoice‑only customers (Auto_Collection__c = false) → a hosted Pay Link is generated for the customer to pay.

A successful Chargent Transaction flows back: a Payment is created and applied to the invoice (Collection_Status = Paid / Partially Paid). A failed charge sets Collection_Status = Failed and is retried by the dunning engine (due+3, due+6) before ending at due+61.

How to use this guide

Do the one‑time Test‑data setup first, then work each area. Each area lists the steps, shows what the real record looks like, and states the pass criteria. Finance follows the reconciliation checks.

⚙ Test‑data setup no migration needed

The only thing migration normally provides that UAT needs is a stored card token on the account. You create one by tokenizing an Auth.net sandbox card. ~10 minutes, once.

  1. Create / pick a test Account (e.g. "UAT – Auto Collect Co"). Make two — one auto‑collect, one invoice‑only — to cover both paths.
  2. Tokenize a sandbox card. Create a Charge Order on the account (Gateway = Authorize.net Test, an Opportunity, Payment Method = Credit Card), open the Chargent payment component, New Payment Method → enter the test card (see below) → tokenize. This creates an Auth.net CIM token like 526956659‑539030077.
  3. Copy the token to the Account field Chargent_Payment_Token__c. (This is the step migration automates.)
  4. Set the collection mode: Auto_Collection__c = TRUE for auto‑charge, or FALSE for the Pay‑Link path. Optionally set Apply_Surcharge__c.
  5. Generate an invoice (Bill Now on an activated Order, or an existing Draft), set its Opportunity__c, and Post it.
Heads‑up on the Account fields
The collection fields (Apply_Surcharge__c, Auto_Collection__c, Chargent_Payment_Token__c) are on the standard Account layout. Some accounts use a record‑type‑specific (managed) layout where these aren't shown — set those via the API/data loader, or have an admin add the fields to that layout. Field security is already granted via Groove_Invoice_Admin.

Auto‑collect

1 Auto‑charge a stored‑card customer ✓ Proven (real charge)

Goal

A Posted invoice on an auto‑collect account is charged automatically end‑to‑end, with the payment applied back and the invoice marked Paid.

Steps

  1. On an auto‑collect account (token set, Auto_Collection = true), open a Draft invoice with an Opportunity and a balance.
  2. Click Post. (Posting is asynchronous here — allow a few minutes for Posting In Progress → Posted.)
  3. Watch the bridge run automatically — no further clicks.
Expected result
A Charge Order is auto‑created and charged → an Approved Transaction → a Payment applied → invoice Balance 0, Collection Status = Paid.
Auto‑collect · failure path

2 Declined charge → failure capture → dunning ✓ Proven

Goal

A failed charge is captured cleanly (no phantom payment), the invoice is flagged, and it enters the dunning queue for automatic retries.

Steps

  1. On an auto‑collect account, use a decline sandbox card/trigger when tokenizing.
  2. Post the invoice and let the auto‑charge run.
Expected result
Transaction Declined/Error; no Payment; balance unchanged; invoice Collection Status = Failed with the gateway error in Last Collection Error; invoice now in the dunning set.
Why this matters
A pre‑existing bug let a receipt automation roll back the Transaction — a card could be charged with no Salesforce record. That's fixed: failures now always record (as above), so Finance never has a "charged‑but‑not‑recorded" gap.
Invoice‑only

3 Hosted Pay‑Link for invoice‑only customers ✓ Proven (live payment form)

Goal

For customers who pay themselves, the bridge generates a branded hosted payment link on the invoice instead of auto‑charging.

Steps

  1. On an invoice‑only account (Auto_Collection = false), post an invoice.
  2. Open the invoice's Pay Link and pay it to confirm the round‑trip.
What works — link generation ✓
Charge Order + Payment Request created; invoice Pay Link populated; Collection Status = Manual / Invoice Only.
What works — the customer payment page ✓
Opening the Pay Link (freshly generated, logged‑out browser) renders the Groove‑branded payment form — card number, name, expiry, CVV, billing address, and Confirm & Pay $174.00. Verified live 2026‑06‑29.
Fixed: a sandbox‑refresh config gotcha (important for refreshes & cutover)
Links initially showed "expired" because the Payment Request config's Salesforce Sites Page URL still pointed at the production domain (groove.my.salesforce-sites.com) — a sandbox‑refresh leftover — so sandbox tokens were validated against production. Fixed by pointing it at the sandbox domain (groove--fc.sandbox.my.salesforce-sites.com). This URL is environment‑specific: re‑set it after every sandbox refresh, and keep the production value in production (don't let the sandbox value promote to prod).

Dunning

5 Dunning / retry engine Time‑gated

Goal

Failed invoices are automatically retried and, if still unpaid, closed out — with the subscription left active.

The scheduled flow Invoice_Dunning_Retry runs daily (08:00 UTC) over Posted / unpaid / Failed invoices. The tracking fields live in the invoice's Collection section (Dunning Retry Count, Last Dunning Attempt — visible in the UAT‑1 screenshot).

Expected behaviour over time
due+3 → retry #1 (count → 1) · due+6 → retry #2 (count → 2) · due+61 → Collection Status = Dunning Ended - Not Paid (subscription left active). A retry that succeeds → Paid.
Testing note
DueDate is immutable on Posted invoices, so you can't fast‑forward it. Either let a real Failed invoice age, or ask SaaScend to seed a time‑shifted Failed invoice (pre‑Post) to exercise a full cycle.

Surcharge

7 Surcharge toggle ✓ Logic proven

Goal

Surcharge is a single manual switch — off for most customers, on for the few who get surcharged.

Steps

  1. On a test account set Apply Surcharge = TRUE (default is OFF).
  2. Post an invoice / create a Charge Order for that account.
  3. Open the Charge Order and check the Waive Fee field.
Expected result
Apply_Surcharge = TRUE → Charge Order Waive Fee = false (surcharge applies). FALSE → Waive Fee = true (waived). Verified both directions in sandbox.
One dependency
Salesforce only toggles Waive Fee; the surcharge rate is configured in Chargent. Until Chargent's fee feature is set with a rate, the toggle flips correctly but no surcharge amount is added.
